### 1.1 Project Goals

**Primary Objective:**
Develop a comprehensive web-based information management system for pineapple entrepreneurs to manage their business information, farms, shops, and connect with customers through announcements and an AI-powered chatbot.

**Key Goals:**
1. Create a unified platform for pineapple entrepreneurs to manage their business profiles
2. Enable entrepreneurs to showcase farms and shops with interactive maps
3. Provide announcement system for prices, promotions, roadshows, and news
4. Implement AI chatbot for customer support using Google Gemini API
5. Build admin panel for system management and entrepreneur approval workflow
6. Ensure responsive design for mobile and desktop access

### 1.4 STARR Reflection - Project Scope

**Situation:**
The project began with a clear vision to create a comprehensive information management system for pineapple entrepreneurs. Initial scope included core features like user management, farm/shop management, announcements, and a chatbot.

**Task:**
Define and maintain project scope throughout the development lifecycle, ensuring all team members understood deliverables and boundaries.

**Action:**
- Conducted detailed requirements gathering sessions with stakeholders
- Created comprehensive WBS breaking down the project into manageable tasks
- Established clear in-scope and out-of-scope features
- Documented all features in PRD (Product Requirements Document)
- Conducted regular scope reviews during sprint meetings

**Result:**
- Successfully delivered all planned in-scope features
- Maintained focus and avoided scope creep
- Clear understanding among team members about project boundaries
- All deliverables met initial project goals

**Reflection:**
The project scope aligned well with initial expectations. The WBS provided excellent structure and helped prevent scope creep. However, we learned that:
- **Positive:** Clear scope definition early prevented confusion and rework
- **Challenge:** Some features (like price scraping) required more time than initially estimated
- **Improvement:** Future projects should include buffer time for complex integrations
- **Learning:** Regular scope reviews helped identify potential issues early

### 3.2 Risk Mitigation Strategies Detail

#### High Priority Risks

**R001 - API Key Limitations:**
- **Mitigation Actions Taken:**
  - Implemented caching for Google Maps API calls
  - Set up API usage monitoring dashboard
  - Created fallback mechanisms for when API is unavailable
  - Documented API key rotation procedures

**R004 - Security Vulnerabilities:**
- **Mitigation Actions Taken:**
  - All database queries use PDO prepared statements
  - Input sanitization using Helper::sanitize()
  - File upload validation (type, size, content)
  - Session security measures implemented
  - Regular code reviews focused on security

**R008 - Integration Complexity:**
- **Mitigation Actions Taken:**
  - Created separate helper classes for each API integration
  - Implemented comprehensive error handling
  - Used mock responses during development
  - Phased integration (Maps first, then Gemini)

### 3.4 STARR Reflection - Risk Management

**Situation:**
The project involved multiple external API integrations, complex features, and tight deadlines, creating various potential risks that could impact project success.

**Task:**
Identify, assess, and manage all potential risks throughout the project lifecycle to minimize negative impacts on deliverables and timeline.

**Action:**
- Conducted risk identification workshop with entire team
- Created comprehensive risk register with 12 identified risks
- Assessed each risk for probability and impact
- Developed specific mitigation strategies for each risk
- Assigned risk owners for monitoring
- Conducted weekly risk reviews
- Updated risk status regularly

**Result:**
- All identified risks were successfully mitigated
- No major risks materialized that significantly impacted the project
- Early identification of API integration challenges allowed proactive solutions
- Security measures prevented any security incidents
- Project completed on time with quality deliverables

**Reflection:**
The risk management process was effective in preventing issues. Key learnings:
- **Positive:** Early risk identification allowed proactive mitigation
- **Positive:** Regular risk reviews helped catch emerging issues early
- **Challenge:** Some risks (like API integration complexity) required more effort than initially estimated
- **Success:** Security-focused approach prevented vulnerabilities
- **Learning:** Risk management should be ongoing, not just at project start
- **Improvement:** Could implement automated risk monitoring for technical risks (API quotas, performance)
